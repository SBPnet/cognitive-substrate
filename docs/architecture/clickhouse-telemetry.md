# ClickHouse Telemetry Layer

## Role in the Storage Architecture

The cognitive-substrate storage model assigns each datastore a distinct responsibility:

| Store | Role |
|---|---|
| PostgreSQL | Canonical configuration, service topology, alert policies |
| OpenSearch | Semantic and associative memory, pattern library, vector retrieval |
| ClickHouse | Temporal telemetry intelligence: raw metrics, logs, traces, cognitive events, reinforcement signals |

ClickHouse fills the gap between semantic cognition and high-volume operational telemetry. It is the append-only analytical engine for all time-series data generated by the 1 000+ Aiven services the system monitors. OpenSearch is not suitable for this workload because it is optimised for document retrieval and vector search, not aggregation-heavy analytical scans over hundreds of billions of rows.

## Table Inventory

All tables reside in the `cognitive_substrate_telemetry` database on the Aiven ClickHouse service provisioned by `infra/aiven/clickhouse.tf`.

### `metrics_raw`

Raw metric data points ingested from Aiven service integrations and OTEL collectors. Partition key `toYYYYMMDD(timestamp)`, order by `(environment, service_type, service_id, metric_name, timestamp)`, 7-day TTL. This is the hot tier: high write throughput, short retention, used for real-time anomaly detection and pattern seed.

### `logs_raw`

Structured log records from all monitored services. Same partition key and retention as `metrics_raw`. Indexed by `(environment, severity, service_type, service_id, timestamp)`.

### `traces_raw`

OTEL span data. 3-day TTL because trace volume is highest and spans are most useful immediately after an incident. Indexed by `(environment, status, service_id, operation_name, timestamp, trace_id)`. Exact trace reconstruction uses `trace_id` after the relevant environment and incident window have been identified.

### `cognitive_events`

Operational primitive signals produced by the telemetry worker after normalisation. Long-term retention (no TTL by default). Indexed by `(environment, primitive_id, timestamp)`. This is the primary learning substrate: pattern and reinforcement engines read from this table to accumulate evidence.

### `pattern_outcomes`

Reinforcement feedback records written by the reinforcement worker. Each row links a recommendation to an observed outcome and records the confidence update applied to the pattern. Indexed by `(environment, pattern_id, timestamp)`.

### `incident_reconstruction`

Full incident timeline rows for replay. Every event (metric, primitive, pattern match, recommendation, operator action) related to an incident is written here with a shared `incident_id`. Indexed by `(environment, incident_id, timestamp)`. This enables cognitive replay: reprocessing historical incidents through updated models.

## Retention Strategy

```
Hot tier (raw telemetry)
  metrics_raw  : 7 days   (sufficient for anomaly detection + pattern seed)
  logs_raw     : 7 days
  traces_raw   : 3 days

Warm tier (cognitive signal)
  cognitive_events : indefinite  (learning substrate)
  pattern_outcomes : indefinite  (reinforcement history)

Replay tier
  incident_reconstruction: indefinite
```

Long-term cognitive and replay tables are small relative to the raw tiers because only detected signals and matched patterns are stored there, not every raw metric point.

## Write Path

```
Aiven services
    ↓ (OTEL collector / Aiven metrics integration)
telemetry.metrics.raw  (Kafka, Diskless topic)
    ↓
telemetry-worker
    ├── INSERT metrics_raw  (raw preservation)
    └── INSERT cognitive_events  (after normalisation)
        ↓
        telemetry.events.normalized  (Kafka)
        cognition.primitives         (Kafka)
```

The Aiven Kafka-to-ClickHouse service integration (`infra/aiven/clickhouse.tf`) provides an alternative direct path for high-volume raw rows without the telemetry worker. The worker is still responsible for the cognitive_events table because it applies system-specific normalisation.

The typed inserter batches rows by table, chunks large arrays at 100 000 rows per request, and exposes strict-mode validation for batches below 1 000 rows. The ClickHouse client enables `async_insert=1`, `wait_for_async_insert=1`, `async_insert_busy_timeout_ms=1000`, and `async_insert_max_data_size=10000000`, which allows high-frequency small batches while preserving durability feedback.

## Query Patterns

All queries are aggregation-heavy, time-windowed, and append-only:

```sql
/* What primitives have been most active in the last 15 minutes? */
SELECT primitive_id, avg(intensity) AS avg_intensity, count() AS count
FROM cognitive_events
WHERE environment = 'prod'
  AND primitive_id IN ('latency_spike', 'error_burst')
  AND timestamp > now() - INTERVAL 15 MINUTE
GROUP BY primitive_id
ORDER BY avg_intensity DESC;

/* What is the success rate of recommendations by pattern? */
SELECT
    pattern_id,
    countIf(outcome = 'success') / count() AS success_rate,
    avg(confidence_after - confidence_before) AS avg_confidence_delta
FROM pattern_outcomes
WHERE environment = 'prod'
  AND timestamp > now() - INTERVAL 30 DAY
GROUP BY pattern_id;
```

The dominant query patterns are documented beside each DDL constant in `packages/clickhouse-telemetry/src/schemas.ts` because ClickHouse `ORDER BY` choices are immutable after table creation.

## Relationship to OpenSearch

ClickHouse and OpenSearch are complementary, not overlapping:

- ClickHouse answers: what happened, how fast, how often, what correlates.
- OpenSearch answers: what does this resemble, what happened before, what fixed similar issues.

Raw telemetry is never indexed in OpenSearch. Only compressed cognitive artifacts (pattern documents, incident summaries, embeddings) reach OpenSearch from the telemetry pipeline.
