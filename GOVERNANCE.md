# Governance

Cognitive Substrate is an open-core infrastructure project. Core runtime packages, schemas, reference deployments, and research artifacts are developed in the open. Managed operations, hosted governance, multi-tenant orchestration, and advanced observability products may be developed as commercial extensions.

Project maintainers are responsible for accepting changes, preserving architectural coherence, and enforcing the written-content rules in `docs/style-guide.md`.

## Maintainer Responsibilities

- Review changes for architectural coherence, security impact, and documentation accuracy.
- Keep package boundaries aligned with the open-core strategy.
- Require evidence for performance, reliability, and transfer claims.
- Maintain release notes, migration notes, and supported-version guidance.

## Contribution Workflow

Changes that modify public APIs, event contracts, schemas, or deployment topology should include an architecture note or design rationale. Changes that affect runtime behavior should include tests or an explicit test gap. Documentation changes under `docs/` must follow `docs/style-guide.md`.

## Release Process

Releases should move through research preview, local runtime, reference deployment, hosted alpha, and enterprise governance alpha milestones. Each release should publish verification status, known limitations, and migration guidance.

## Decision Principles

- Preserve infrastructure credibility.
- Avoid AGI, sentience, consciousness, and biological-equivalence claims.
- Keep public APIs stable once documented.
- Prefer observable, bounded adaptation over opaque self-modification.
- Treat benchmark evidence as required for quantitative claims.

## Trademark

The Cognitive Substrate name identifies the project and its ecosystem. Community use should describe compatibility or contribution without implying official endorsement unless maintainers have granted that status.
